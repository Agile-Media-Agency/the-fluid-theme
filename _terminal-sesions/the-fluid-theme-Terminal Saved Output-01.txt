⏺ My call: both raw + minified (src/ and dist/fluid.min.css).

  Here's why it fits the project:
  - Raw source stays in src/ — readable, customizable, true to the "no build step" philosophy
  - Minified dist/ is what CDN users point to in production (<link href="...fluid.min.css">)
  - It's what Bootstrap, Bulma, and every major CSS framework does
  - jsDelivr and unpkg will auto-serve either file; users choose based on need

  I have enough to propose approaches now.

  ---
  Three approaches for the npm publish setup:

  Option A — Manual publish workflow (recommended)
  Set up the npm account + @fluid-theme org manually, add a build script that generates
  dist/fluid.min.css using lightningcss (fast, modern CSS minifier), update package.json with the scoped
  name and correct files, then publish manually with npm publish --access public. Simple, you're in
  control, no CI secrets needed yet.

  Option B — Publish + GitHub Actions release automation
  Same as A but wire up the existing .github/ release workflow to auto-publish to npm on version tags.
  More setup upfront, but future releases are one git tag away.

  Option C — Publish raw CSS only, skip the build step
  No minification, just update the package name and publish. Simplest possible path — but CDN users get
  unminified CSS and you miss the production-ready story.

  My recommendation: Option A. Get the package live quickly, understand the process end-to-end, then add
  automation (Option B) once you know it works. Option C leaves CDN users with a worse experience.

  Does Option A sound right to you?

❯ should my npm account be a "top level" ie agile media agency (my llc) for all projects that might use  
this or domain/project level, ie: npm@thefluidtheme.com etc                                              

⏺ One npm account, multiple organizations. Here's why the distinction matters:

  npm account = your login identity (one per person/company, used to authenticate)
  npm organization = the package scope (@fluid-theme, @agile-media, etc.)

  A single npm account can own as many organizations as you want. So:

  Recommended: one top-level account — something like neilwhite or agilemediaagency — and then create
  separate orgs per project:

  ┌──────────────┬─────────────────────────────────────────────┐
  │     Org      │                  Packages                   │
  ├──────────────┼─────────────────────────────────────────────┤
  │ @fluid-theme │ @fluid-theme/core, @fluid-theme/react, etc. │
  ├──────────────┼─────────────────────────────────────────────┤
  │ @agile-media │ any future LLC tools/packages               │
  └──────────────┴─────────────────────────────────────────────┘

  Don't use an email-style name like npm@thefluidtheme.com — that's not what npm account names are for.
  npm usernames are short handles, not emails (though you register with an email address).

  My recommendation: Register as neilwhite (or agile-media if you prefer the LLC brand). Then create the
  @fluid-theme org from that account. If you ever have collaborators, you can add them to specific orgs
  without giving them your login.

  The account name is mostly invisible to consumers of your packages — they only ever see
  @fluid-theme/core, never who the account owner is.
